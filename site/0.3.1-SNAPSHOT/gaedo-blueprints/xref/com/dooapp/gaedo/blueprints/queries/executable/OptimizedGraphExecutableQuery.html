22  /**
23   * Optimized executable query.
24   * How is it optimized ? Instead of simply searching through all vertices linked to class vertex (which always exist) through
25   * Object.classes edge, this one tries to find the vertex linked to the smallest solution space. How ?
26   * As an example, suppose one want to find, using our test beans, elements matching that query :
27   * <pre> 
28   *  AND
29   *	  Posts.note ==? 4.0
30   *	  Posts.author.login ==? "user login"
31   *	  Object.classes contains Post.class
32   * </pre>
33   * This class will start by extracting query roots (vertices to which solutions of this query MUST be linked) by calling {@link #getPossibleRootsOf(VertexTest)}
34   * In this example, this method call will return
35   * <pre> 
36   * 4.0:long => { Property[note] }
37   * "user login":string => { Property[author], Property[login] }
38   * Post:class => { Property[class] }
39   * </pre>
40   * One question remain : from these three vertices, which one will be linked (through the given property path) to the smallest number of vertices ?
41   * To answer this question, the {@link #findBestRootIn(Map)} method creates a {@link VertexValueRange} object which can back-navigate the edges representing these properties
42   * and directly give us the best query root.
43   * 
44   * @author ndx
45   * 
46   */
